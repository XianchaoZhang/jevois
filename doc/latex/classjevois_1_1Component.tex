模型层次结构的组成部分。 

模型 Components 构成了每个 Je\+Vois 应用程序的核心，因为它们的主要目的是 1) 从共享的、可重用的处理元素构建复杂的视觉处理管道； 2) 这些处理元件可能有可调参数的统一接口。事实上，每个需要向用户公开参数的类都应该直接或间接继承自 \mbox{\hyperlink{classjevois_1_1Component}{Component}} 。参数是任何有效 C++ 类型的简单包装，以及一些描述、默认值、可能的有效值规范等。参数用于通过更改参数值来调整处理元素的行为（例如，某些图像处理 算法中的阈值）。

Components 通常形成一个组件层次结构，其中 \mbox{\hyperlink{classjevois_1_1Manager}{Manager}} （包括 \mbox{\hyperlink{classjevois_1_1Engine}{Engine}} ）位于根，下面是子组件树。要形成这棵树，只需使用 \mbox{\hyperlink{classjevois_1_1Manager_ac5e322bf63b28a786e73812541bc4508}{Manager\+::add\+Component()}} 和 \mbox{\hyperlink{classjevois_1_1Component_a8348eba2c31efd4244afc97b25ea6fcc}{Component\+::add\+Sub\+Component()}} 方法添加一些子项，然后向这些子项添加一些子项，等等...

\mbox{\hyperlink{classjevois_1_1Component}{Component}} 背后的目的之一是在实现复杂的视觉处理框架时实现大量代码重用，并减轻在调用处理元素时将许多参数传递给函数调用的负担。这 与 Open\+CV 等许多其他框架有很大的不同，后者基本上依赖于有许多参数的函数（尽管后来的版本也使用了类）。例如，在 Open\+CV 中调用 Canny 边缘检测器的方法是调用以下函数：


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keywordtype}{void} Canny(InputArray image, OutputArray edges, \textcolor{keywordtype}{double} threshold1, \textcolor{keywordtype}{double} threshold2,}
\DoxyCodeLine{           \textcolor{keywordtype}{int} apertureSize = 3, \textcolor{keywordtype}{bool} L2gradient = \textcolor{keyword}{false})}
\end{DoxyCode}


除了可能会混淆哪个值属于长列表中的哪个参数（\+Python 等语言通过允许按名称访问参数来解决）之外，这种方法的一个主要问题是每个使用 Canny 的函数都必须为用户提供一种机制设置参数（threshold1, threshold2 等），或者在大多数情况下，这些参数最终会被硬连线，限制最终应 用程序对不同图像尺寸、环境类型等的适用性。

相比之下，在 Je\+Vois 框架中，我们将创建一个 Canny \mbox{\hyperlink{classjevois_1_1Component}{Component}} ，其中包含阈值等参数设置以及一个成员函数


\begin{DoxyCode}{0}
\DoxyCodeLine{OutputArray process(InputArray in)}
\end{DoxyCode}


通常首先设置参数值，然后调用该函数。设置参数可以通过使用该函数的代码来完成，但更常见的是由用户完成。在特定的视觉管道中，可以在开始 时为参数提供合理的默认值，然后让那些可能想要修改它们的最终用户可以访问这些参数。\+Je\+Vois 中参数的修改可以在应用程序启动时通过解析命 令行参数、实例化新的处理 \mbox{\hyperlink{classjevois_1_1Module}{Module}} 时进行处理，也可以在其运行时通过 \mbox{\hyperlink{classjevois_1_1Serial}{Serial}} 端口与管理系统的 \mbox{\hyperlink{classjevois_1_1Engine}{Engine}} 交互来处理。

任何在其层次结构中包含 Canny 组件的复杂视觉处理管道都会公开其参数，以便可以由用户（通过命令行参数、通过串行端口发出的命令等）或层次结构 中的其他组件来设置它们。如果层次结构中存在 \mbox{\hyperlink{classjevois_1_1Component}{Component}} 的多个实例，则它们的参数可以一次全部设置为单个（共享）值，也可以单独访问它们并将其 设置为不同的值（每个 \mbox{\hyperlink{classjevois_1_1Component}{Component}} 都有唯一的实例名称，并且所谓的描述符是通过连接组件的实例名称和特定 \mbox{\hyperlink{classjevois_1_1Component}{Component}} 实例的所有父参数来创建的； 可以通过将该实例的描述符添加到参数名称来访问。参数提供了用于设置/获取参数值的统一接口，因此程序员不必在每个组件中为每个参数编写访问器函数 （例如，添加 threshold1 Parameter$<$int$>$，会自动将 set()、get() 等函数添加到你的组件中）。

此外，\+Component 还提供某种程度的内省（introspection）或反射（reflection）：可以访问组件实例名称、类名称、描述符（包括层次结构中所有父 组件的名称），并迭代其参数、子组件等。

\mbox{\hyperlink{classjevois_1_1Component}{Component}} 本质上是线程安全的，组件接口的编写是为了鼓励线程安全的使用；例如，没有提供返回组件父组件的函数，因为另一个线程可能会在返回 父组件的时间和使用它的时间之间更改该组件。相反，提供的函数可以以线程安全的方式向上或向下遍历层次结构，并对遍历期间遇到的组件执行某些 操作。

组件分几个阶段使用：


\begin{DoxyItemize}
\item Construction\+: 在构造函数中，参数值尚未设置，因此依赖于它们的任何内容都不应初始化。通常，构造函数应该只初始化独立于参数值的固定组件 资源，或者它们也可以设置参数的默认值，这些默认值稍后可能会被命令行参数覆盖。在许多情况下，构造函数不会执行任何操作，并且大部分初始化 将延迟到组件的 init() 阶段设置参数值。
\item Initialization\+: 当在管理器（包括 \mbox{\hyperlink{classjevois_1_1Engine}{Engine}} ）上调用 init() 时，它会向下传播整个 \mbox{\hyperlink{classjevois_1_1Component}{Component}} 层次结构，如下所示：
\begin{DoxyItemize}
\item 在所有子组件上递归调用 \mbox{\hyperlink{classjevois_1_1Component_ace4f9ea2f56c740d429de7ac0b89e7e6}{pre\+Init()}}。
\item 在当前组件上运行 \mbox{\hyperlink{classjevois_1_1Component_ace4f9ea2f56c740d429de7ac0b89e7e6}{pre\+Init()}}，pre\+Init()函数可以由派生类重写以实现一些自定义初始化；请注意，参数值此时尚未最终确定，不应使用）。
\item 然后，\+Manager 解析命令行并可能设置参数。从此时起，认为参数可以使用。
\item 对于有回调但命令行中没有显式设置的参数，请使用默认参数值第一次调用回调。这样，创建资源的回调（例如，设备的设备名称参数的回调， 在调用时打开设备）在 init() 完成之前至少调用一次。
\item 递归地将所有子组件的状态翻转为已初始化，然后将组件状态翻转为已初始化。
\item 在所有子组件上递归运行 \mbox{\hyperlink{classjevois_1_1Component_ae33c39b7b433a471c76f841a725aca8c}{post\+Init()}} （派生类可能会重写 \mbox{\hyperlink{classjevois_1_1Component_ae33c39b7b433a471c76f841a725aca8c}{post\+Init()}} ，并且通常会根据重写 \mbox{\hyperlink{classjevois_1_1Component_ae33c39b7b433a471c76f841a725aca8c}{post\+Init()}} 中的参数值来分配资源）， 然后在当前组件上运行。
\end{DoxyItemize}

初始化流程是固定的。从 \mbox{\hyperlink{classjevois_1_1Component}{Component}} 派生的类可以重写 \mbox{\hyperlink{classjevois_1_1Component_ace4f9ea2f56c740d429de7ac0b89e7e6}{pre\+Init()}} 来实现不依赖于参数值的初始化，并重写 \mbox{\hyperlink{classjevois_1_1Component_ae33c39b7b433a471c76f841a725aca8c}{post\+Init()}} 来实现依赖于 参数值的初始化。一旦 \mbox{\hyperlink{classjevois_1_1Manager}{Manager}} 上的 init() 完成，层次结构中的所有组件都被视为准备好运行。

如果将组件添加到已初始化的层次结构中，则在添加时它将进入与其父组件相同的初始化状态。当从已经运行并流式传输视频的 Je\+Vois 系统 上的 U\+SB 主机选择新的视频格式时，通常会出现这种情况。处理旧格式的 \mbox{\hyperlink{classjevois_1_1Module}{Module}} 首先由 \mbox{\hyperlink{classjevois_1_1Engine}{Engine}} 去初始化，然后将其销毁，为新格式实例化 新模块，并在视频帧流定向到它之前对其进行初始化。
\item uninit\+: 这是 init() 的对偶， \mbox{\hyperlink{classjevois_1_1Component_aa2389e9024d8157b3025bced8bb6c95c}{pre\+Uninit()}} 和 \mbox{\hyperlink{classjevois_1_1Component_abd9cafdcc5e9981b72130aeb3d9ae83f}{post\+Uninit()}} 遵循与 \mbox{\hyperlink{classjevois_1_1Component_ace4f9ea2f56c740d429de7ac0b89e7e6}{pre\+Init()}} 和 \mbox{\hyperlink{classjevois_1_1Component_ae33c39b7b433a471c76f841a725aca8c}{post\+Init()}} 相同的逻辑。具体顺序是：
\begin{DoxyItemize}
\item \mbox{\hyperlink{classjevois_1_1Component_aa2389e9024d8157b3025bced8bb6c95c}{pre\+Uninit()}} 在所有子组件上递归调用，然后在组件上调用（请注意，此时 \mbox{\hyperlink{classjevois_1_1Component_a465314997843a6364a18abce2d307be9}{initialized()}} 仍然为 true）；
\item 组件翻转为未初始化，然后递归其所有子组件
\item \mbox{\hyperlink{classjevois_1_1Component_abd9cafdcc5e9981b72130aeb3d9ae83f}{post\+Uninit()}} 在组件上调用，然后在其所有子组件上递归调用。
\end{DoxyItemize}
\end{DoxyItemize}

请注意 init() 和 uninit() 在 \mbox{\hyperlink{classjevois_1_1Component}{Component}} 中都是私有的，因此不能直接调用。\+Manager（以及从 \mbox{\hyperlink{classjevois_1_1Manager}{Manager}} 派生的 \mbox{\hyperlink{classjevois_1_1Engine}{Engine}} ）将它们公开， 以便您可以在管理器上调用它们。也就是说，您的整个层次结构要么已初始化，要么未初始化。

由于 Je\+Vois 硬件平台上电时会自动构建并初始化默认处理管道，并且当用户更改视频格式时 \mbox{\hyperlink{classjevois_1_1Engine}{Engine}} 会实例化并初始化处理模块，因此大多数 组件只是假设在组件未初始化时它们的视觉处理函数永远不会被调用。因此完全跳过对 \mbox{\hyperlink{classjevois_1_1Component_a465314997843a6364a18abce2d307be9}{initialized()}} 的测试（以节省 C\+PU 周期）。

Pseudo-\/constructor\+: 构造并添加另一个组件作为该组件的子组件。 

将创建 Comp 类型的子组件（必须从 \mbox{\hyperlink{classjevois_1_1Component}{jevois\+::\+Component}} 派生）并将其添加为该组件的子组件。子组件在逻辑上 \char`\"{}属于\char`\"{} 该组件，并且 会随着父组件自动初始化、去初始化和删除。除了构造之外，添加子组件还将使其达到与所有者组件相同的 init() 状态。

Set a parameter value. 

如果我们没有该名称的参数或者该值不起作用，则会抛出异常。以下是描述符的工作原理：

基本格式是


\begin{DoxyCode}{0}
\DoxyCodeLine{[ComponentInstanceName]:[...]:[paramname]}
\end{DoxyCode}


请注意，{\ttfamily \mbox{[}Component\+Instance\+Name\mbox{]}} 标记也可以替换为 $\ast$，这相当于任意数量的任何 Component\+Instance\+Name 规范。 因此，当您不知道参数有多深时，请使用 $\ast$ 来获取参数。如果 $\ast$ 和最终参数名称之间存在其他 Component\+Instance\+Name，则递归将关闭， 并且 $\ast$ 和参数名称之间的任何内容都必须与介入的组件/实例相匹配。

For example,


\begin{DoxyCode}{0}
\DoxyCodeLine{MyInst:*:CoolComp:MyParam}
